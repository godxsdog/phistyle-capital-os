# Ticket: PW-2 r3 — 錢包資訊架構重設計(來源計畫視角)

FABLE-APPROVED: yes (2026-07-07)。OWNER: Codex。VERDICT: new Fable
session。frontend 為主 + 少量 additive API;禁改 schema(需要即
STOP)。使用者驗收 = 唯一標準。

## 設計哲學(BINDING)

使用者的心智模型 = 他的 Excel:一個來源計畫一頁,每頁是該計畫
的完整世界。不是 CRUD 資料表。打掉現有 /wallet 頁面結構重排。

## 導覽(左側或頂部,大字繁中)

總覽 | 萬里通 | 萬豪 | 吉祥 | 其他來源 | 我的點數 | 換票比價

「萬里通/萬豪/吉祥」= from_program 有規則或被釘選的計畫,各自
一頁;「其他來源」收其餘 from_program,可一鍵升格成獨立頁
(釘選,存 localStorage)。

## 來源計畫頁版型(三段式,每頁相同)

1. 頂部卡片:我的這個計畫點數成本(每點 NT$,可就地編輯,
   來源=我的 cost lots 或手動輸入)+ 目前餘額(凱章/老婆並列)。
2. 中段:完整兌換表(=他 Excel 那張表):每列 = 目標計畫|比例
   (送X:得Y)|加贈%|門檻規則(如「每滿60,000送5,000」人話顯示)
   |試算欄(輸入目標點數,即時顯示要送多少+成本多少台幣)|
   查證連結|編輯|刪除。列內就地編輯,不跳頁。空表時顯示
   「+新增兌換規則」大按鈕與說明。
3. 底部:快速試算器:「我要 [目標計畫▼] [數量] 點」→ 即時顯示
   需送多少本計畫點數、經哪條規則、總成本台幣(呼叫既有引擎
   read-only API 或前端算)。

## AI 解析貼上文字(取代「自動抓比例」,BINDING 邊界)

- DeepSeek 沒有瀏覽網頁能力,直接「上網撈比例」會產生幻覺數字
  = 禁止。合法替代:每個來源頁提供「AI 解析」框:使用者把促銷
  頁/公告文字貼進來 → 後端呼叫 DeepSeek(Phase 15 呼叫模式,
  嚴格 JSON)解析出結構化規則(from/to/比例/加贈/期限)→ 顯示
  為「待確認」預覽卡 → 使用者按「確認加入」才入庫,note 標
  「AI解析-已人工確認」。LLM 失敗 → 顯示「解析失敗,請手動輸入」。
- 絕不自動入庫、絕不背景爬取。

## 我的點數頁

保留 r3 的凱章/老婆結構(那版方向是對的):各帳戶餘額、成本、
到期,新增交易入口在每列旁邊(不是獨立大表單)。

## 換票比價頁(/wallet/awards 重排)

輸入表單一行式:「我要 [計畫▼] [哩數] 哩,稅金 [金額][幣別▼],
現金票價 [NT$](選填)」→ 大按鈕「幫我比價」。結果表:排名|
方案(人話:「用凱章既有萬豪點,經 60k 分段轉 AS」)|總成本|
每點成本|剩餘點數|明細展開。第一名加皇冠標記與「比現金票省
NT$X」大字。
