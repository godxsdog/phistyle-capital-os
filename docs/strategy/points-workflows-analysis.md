# 使用者原始點數工作流分析(Fable, 2026-07-07)

來源:使用者親述的三套真實流程(平安萬里通/Marriott 飯店兌換/
Marriott→航空)。指令:先理解,不重設計、不寫碼、不開新 phase。
本文件是未來所有 PW 決策的上位參照。

## 1. 共用決策邏輯(三套流程是同一個演算法)

目標先行(先確認票/房真實存在與需求)
→ 缺口計算(所需 − 現有 = 真實缺口,只補缺口)
→ 枚舉所有資金路徑(既有/轉點鏈/買分/信用卡/現金)
→ 每條路徑換算到同一終點單位的真實成本
   (bonus 一律進分母;門檻一律分批算;層層 bonus 疊加後
   只除以最終實際入帳數)
→ 機會成本(多用途資產如 Marriott:住宿價值 vs 轉航空價值,
   兩條路互為對方的成本)
→ 總補足成本最低者勝(不是每點單價最低)
→ 不可逆動作(轉點)永遠最後執行,且前置全部確認才動

四條鐵律:持有≠該用(拒絕沉沒成本);轉點不可逆所以票先確認;
贈點進分母;有期限資產(FNC/bonus 活動)優先處理。

## 2. 重複人工步驟(自動化候選)

同房同日現金價vs點數價對照查詢;award 放票查詢;逐路徑手算
成本;bonus 活動期限追蹤;FNC 到期追蹤;多帳戶餘額盤點;
雙層 bonus(60k 門檻 × partner 活動)疊算。

## 3. 資料輸入清單

持有:各帳戶餘額、成本批次、FNC(面額+到期日+張數)。
規則:轉點比例、門檻批次、bonus%、活動有效期、最低轉點單位。
市場:award 所需哩數+稅、現金票價、飯店現金價/點數價(逐日)。
匯率:多幣別→TWD。

## 4. 外部比例規則的性質分類

穩定官方(Marriott 60k→25k):一次設定,少維護。
App 內私有(萬里通比例):只能人工維護(已 Excel 萃取 26 條)。
促銷型(partner bonus、買分特賣):有期限,靠 valid_from/until
+ AI 解析貼上文字 + 人工確認。
動態市場(飯店房價、award 票):逐次查詢,不是規則是快照。

## 5. 應成為決定論程式碼的部分(對照現況)

已建成:路徑枚舉與成本排名(PW-2 引擎)、bonus 進分母、
threshold_block 分批、鏈式轉換、唯讀模擬、FIFO 批次成本。
缺口(GAP,未建,列為未來 PW 決策候選,不自動開工):
G1 缺口式混合資金:「既有 18,911 + 只補缺 6,089」——現行引擎
   方法純粹化把這條真實主路徑排除了(PW-2 v0 已知限制)。
G2 FNC 免房券資產類:面額+到期+張數,domain model 完全沒有。
G3 飯店兌換比較器:現金 vs 純點 vs FNC vs FNC+補點(第二個
   evaluator,與機票引擎平行)。
G4 Marriott 機會成本基準:每點住宿價值(使用者設定或由歷史
   兌換記錄推導),供轉航空路徑計入隱含成本。
G5 雙層 bonus 疊加:可用「臨時規則列」的資料方式表達,
   不需新程式;需 UI 引導。

## 6. 真正需要 Brain(人/LLM)判斷的部分

值不值得的門檻(機會成本容忍度=個人品味);促銷條款解讀
(AI 解析輔助+人工確認);放票時機與艙等取捨;貶值風險
(囤 vs 燒);行程偏好。最終轉點 GO/NO-GO = 永遠是人
(對應不變式 1:批准是明確人工行為)。

## 裁定

引擎的數學骨架與使用者邏輯高度吻合;最大結構性缺口是 G1
(缺口式資金)與 G2(FNC 資產類)。是否為此開 PW-4,等使用者
明示,屆時以本文件為規格上位。
